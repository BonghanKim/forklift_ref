30  * @section pIntro Introduction
31  * This work is a part of two projects:
32  * - <a href="http://www.oru.se/English/Research/Research-environments/Research-environment/Centre-for-Applied-Autonomous-Sensor-Systems-AASS/Research/Research-projects/Research-project/?rdb=693">
33  * SAUNA - SAFE AUTONOMOUS NAVIGATION</a>
34  * - <a href="http://www.aass.oru.se/Research/Learning/savie/index.html">SAVIE</a>
35  *
36  * The purpose of this module is to control a car while tracking a certain trajectory.
37  *
38  * The module is adapted for SnowWhite car, which has VMC500 onboard controller that is capable
39  * of executing given steering angles and velocities of the steering wheel. This controller
40  * is produced by Kollmorgen (http://www.ndc8.com/ndc8.html).
41  *
42  * Tracking is realized using model predictive control (MPC). The main purpose of this module is
43  * to form and solve respective quadratic programming (QP) problem. All fromulations are
44  * available in LaTeX format in 'formulations' folder.
45  *
46  * \n
47  *
48  * @section pDepRun Dependencies and running
49  *
50  * @verbinclude "README"
51  *
52  * \n
53  *
54  * @section pDirs Supplementary files and source code
55  *
56  * The purpose of directories in the root of the project:
57  *
58  * - 'config': sample configuration files of the module.
59  *
60  * - 'dummy_node': dummy control node, which was used for integration tests.
61  *
62  * - 'formulations': formulations of the QP problem and its constraints.
63  *
64  * - 'func_gen': a set of functions generated in Maple. They form several matrices, which
65  * are necessary for generation of the QP problem. Each function is generated for a particular
66  * length of the preview window of MPC problem.
67  *
68  * - 'qpOASES': third party QP solver.
69  *
70  * - 'traj_gen': a set of simple Matlab scripts, which are used for generation of test
71  * trajectories and generation of reference QP solutions for tests. 'spline_demo'
72  * subfolder contains an example of smooth trajectory generation with respect to
73  * time constraints. The test trajectories do not take into account initial and final
74  * steering angles (phi), because these angles depend on the curvature of a polynomial,
75  * which is used for interpolation.
76  *
77  * \n

103 /**
104  * @page pIntegration Integration with other modules
105  *
106  * This page contains links to the methods, which communicate with other modules.
107  * Interfaces are described in comments of these methods.
108  *
109  * @section pModuleOut Output of the control module
110  * - In simulation:
111  * - sending commands: commandSender#send;
112  * - trajectory evaluation: trajectoryEvaluator#report.
113  * - On a car:
114  * - trajectory evaluation: trajectoryEvaluator#report.
115  *
116  * Messages from trajectory evaluator are send periodically and can be used as a heartbeat
117  * signal. The delay between two messages depends on the number of trajectories and time
118  * required for solution of a QP. The minimal delay is equal to the control sampling time.
119  *
120  * \n
121  *
122  * @section pModuleIn Input of the control module
123  * - In simulation:
124  * - current state of a car: ROSSensorSubscriber#readLoop;
125  * - trajectory: trajectoryReader#processTrajectoryChunk;
126  * - various commands: trajectoryReader#actOnCommand;
127  * - heartbeat signal: trajectoryReader#checkActivity.
128  * - On a car:
129  * - trajectory: trajectoryReader#processTrajectoryChunk;
130  * - various commands: trajectoryReader#actOnCommand.
131  * - heartbeat signal: trajectoryReader#checkActivity.
132  *
133  * \n
134  *
135  * @section pIntegrationLogic Logic of interaction
136  *
137  * - Transmission of a trajectory
138  * - The sequence number of the first chunk of a trajectory must be equal to 0.
139  * - All following chunks must be submitted in the same order as they appear in
140  * trajectory with appropriately set sequence numbers.
141  * - The last chunk must be explicitly indicated. No chunks are accepted after
142  * the last one.
143  * - If the sequence number of a newly arrived chunk has been already received,
144  * the respective trajectory chunk is replaced by the new one. The new chunk may
145  * contain new constraints and trajectory points.
146  *
147  * - Activation of tracking
148  * - The tracking can be activated, when at least one chunk of one trajectory has
149  * been added.
150  * - It is necessary to activate a particular trajectory with a respective command.
151  * - Then the time, when tracking must start, should be set using a special command.
152  *
153  * - Transmission of trajectories after termination
154  * - Whenever tracking is stopped due to an error or successfuly completed all
155  * trajectories are removed.
156  * - If the goal is reached successfully, the controller enters the initial state
157  * and transmission of new trajectories can be started at once.
158  * - If there was an error, the controller must be switched to a normal state with
159  * a special command explicitly. Otherwise, tracking would not start.
160  */

168 /**
169  * @page pInteraction Interaction with the onboard controller
170  *
171  * There are two methods to communicate with the onboard controller in order
172  * to send commands and receive sensor readings. The first one is telnet connection
173  * via Ethernet cable to Navserver, which runs on the controller. This connection
174  * can be used only to read sensor data. The second one is the CAN bus, which can
175  * be used for sending commands as well. However, the CAN bus baudrate is rather
176  * low, and there is likely to be high delay in receiving the messages from onboard
177  * controller due to high load on the bus.
178  *
179  * Refer to one of these pages for more information on CAN:
180  * - @ref pCANOverview
181  * - @ref pCANOverviewCAN
182  * - @ref pCANOverviewCANopen
183  * - @ref pCANFormat
184  * - @ref pCANSend
185  * - @ref pCANRecv
186  * - @ref pCANStandard
187  *
188  * \n
189  *
190  * @section pCANCable CAN connection
191  * We use a USB cable to connect to the CAN bus on SnowWhite. The cable is produced by Kvaser.
192  * - http://www.kvaser.com/en/developer/canlib.html
193  * - http://www.kvaser.com/canlib-webhelp/
194  *
195  * The Linux version of the driver currently supports only version 2.6 of the kernel and
196  * may have problems with SMP, 64bit and real-time priorities.
197  *
198  * There is one more option to connect to CAN bus (Abdel has used it under Linux) using
199  * hardware and drivers from http://www.peak-system.com/fileadmin/media/linux/index.htm
200  *
201  * \n
202  *
203  * @section pIntRecv Reading data from sensors
204  * The position of the car is updated each 60 ms.
205  *
206  *
207  * @subsection pIntRecvTelnet Connection to Navserver
208  * Connection to the Navserver: address = 192.168.100.100, port = 5432.
209  *
210  * Navserver outputs data in a simple textual format. We can get the necessary data by
211  * subscribing for the 'env' and 'state' updates, which are sent each 60ms. 'env' and
212  * 'state' readings are printed on separate lines. Both lines are sent in one tcp packet.
213  * The interval between reception of two packets varies from 50ms to 70ms.
214  *
215  * @todo There is a a significant delay in execution of requested steering angle (or in
216  * sensor reports), @see stateContainer. It would be nice to know more about operation of
217  * the onboard controller.

292  * @section pIntSend Sending commands
293  * The commands can be sent only via CAN bus. Format of the respective CAN
294  * messages is described in @ref pCANFormat.
295  *
296  * The sampling time of the internal control loop is 20 ms. The execution of
297  * received commands begins on the next iteration of the internal control
298  * loop (20 ms). The flags in the last CAN message with the commands determine
299  * in how many loops the commands are reused.
300  *
301  * \n
302  *
303  * @section pIntSync Synchronization
304  * There are several synchronization options.
305  *
306  * In the first case the MPC controller starts when the data is received from
307  * connection to navserver (see test_01_telnet.cpp).
308  *
309  * In the second case MPC controller is synchronized with CAN messages (id = 128)
310  * since we cannot synchronize with messages with sensor readings. The sensor
311  * readings are obtained from the CAN messages. This approach is more difficult
312  * to implement.
313  *
314  * In the third case MPC controller is also synchronized with CAN messages,
315  * however data is obtained from Navserver. This approach is more difficult to
316  * implement than the first one. The delay between the reception of the data and
317  * the start of the controller is bigger than in the first case.
318  */

321 /**
322  * @page pCANOverview Controller area network (CAN) and CANopen overview
323  *
324  * @section pCANOverviewCAN Controller area network (CAN)
325  *
326  * http://en.wikipedia.org/wiki/Controller_area_network
327  *
328  * CAN bus (for controller area network) is a vehicle bus standard designed
329  * to allow microcontrollers and devices to communicate with each other
330  * within a vehicle without a host computer. CAN bus is a message-based
331  * protocol, designed specifically for automotive applications but now also
332  * used in other areas such as industrial automation and medical equipment.
333  *
334  * CAN is a multi-master broadcast serial bus standard for connecting electronic
335  * control units (ECUs). Each node is able to send and receive messages, but not
336  * simultaneously. A message consists primarily of an ID (identifier), which
337  * represents the priority of the message, and up to eight data bytes.
338  *
339  * If the bus is free, any node may begin to transmit. If two or more nodes
340  * begin sending messages at the same time, the message with the more dominant
341  * ID (which has more dominant bits, i.e., zeroes) will overwrite other nodes'
342  * less dominant IDs. A CAN message that is transmitted with highest priority
343  * will succeed, and the node transmitting the lower priority message will
344  * sense this and back off and wait.
345  *
346  * Message IDs must be unique on a single CAN bus, otherwise two nodes would
347  * continue transmission beyond the end of the arbitration field (ID) causing
348  * an error.
349  *
350  * \n
351  *
352  *
353  * @section pCANOverviewCANopen CANopen
354  *
355  * http://en.wikipedia.org/wiki/Canopen
356  *
357  * Note, that car controller uses CANopen - a communication protocol build on the
358  * top of CAN. Some of the features of CANopen, which are used on the car are
359  * desribed here.
360  *
361  * The Process Data Object (PDO) protocol is used to process real time data among
362  * various nodes. You can transfer up to 8 bytes (64 bits) of data per one PDO
363  * either from or to the device. One PDO can contain multiple object dictionary
364  * entries and the objects within one PDO are configurable using the mapping and
365  * parameter object dictionary entries.
366  *
367  * The Heartbeat protocol is used to monitor the nodes in the network and verify
368  * that they are alive. A heartbeat producer (usually a slave device) periodically
369  * sends a message with the binary function code of 1110 and its node ID
370  * (COB-ID = 0x700 (1800) + node ID). The data part of the frame contains a byte indicating
371  * the node status. The heartbeat consumer reads these messages. If the messages
372  * fail to arrive within a certain time limit (defined in the object dictionary of
373  * the devices) the consumer can take action to, for example, reset the device or
374  * indicate an error.
375  *
376  * http://www.can-cia.org/index.php?id=206
377  *
378  * Synchronization protocol. The Sync Object is implemented with the synchronization
379  * device acting as the producer of that object. In order to guarantee timely access
380  * to the CAN bus the Sync Object is given a very high priority identifier. CANopen
381  * suggests the identifier 128 (0x80) which is a value of the highest priority group
382  * used in the pre-defined connection set. Within the Sync object no application data are
383  * transported. By default, the Sync Object does not carry any data.
384  */
